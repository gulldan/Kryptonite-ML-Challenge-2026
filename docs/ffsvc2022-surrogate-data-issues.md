# FFSVC 2022 Surrogate Data Issues Backlog

## Scope

This note turns the current server-side EDA into an executable cleanup policy for
baseline and training work on `ffsvc2022-surrogate`.

The machine-readable backlog is generated by
[`scripts/data_issues_backlog_report.py`](../scripts/data_issues_backlog_report.py).
It rebuilds the profile, leakage, and audio-quality views from the active
manifests and then classifies each decision as `fix`, `quarantine`, `keep`, or
`document`.

## Current Decisions

## 1. `fix`: resample everything explicitly to `16 kHz`

Evidence:

- active sample-rate mix is `16000=35234`, `44100=27107`, `48000=6206`
- the backlog report classifies `mixed_sample_rates` as a blocking preprocessing
  decision

Decision:

- `KRYP-020` and all downstream feature extraction work must treat explicit
  resampling as part of the preprocessing contract
- backend-default resampling is not acceptable as a source of truth

## 2. `fix`: canonicalize official dev trials that still reference quarantined rows

Evidence:

- the leakage audit now reports `trial_reference_missing`
- two rows in
  `artifacts/manifests/ffsvc2022-surrogate/official_dev_trials.jsonl` still
  point to quarantined duplicate utterances:
  - line `5617` references `ffsvc22_dev_063743.wav`
  - line `34348` references `ffsvc22_dev_002177.wav`

Decision:

- `KRYP-018` should either rewrite official dev trials to canonical utterance ids
  or generate a clean replacement bundle from the active manifests
- official dev trials must be rewritten to canonical utterance ids or filtered
  so every trial row resolves against the active manifests
- threshold tuning or verification benchmarking should not rely on the official
  trial file until this is fixed

## 3. `quarantine`: keep confirmed duplicate-content rows out of the active set

Evidence:

- `artifacts/manifests/ffsvc2022-surrogate/quarantine_manifest.jsonl` contains
  `2` rows
- both rows are tagged as `duplicate_audio_content`
- the canonical active policy is already described in
  [the surrogate dataset note](./ffsvc2022-surrogate.md)

Decision:

- keep `ffsvc22_dev_002177` quarantined in favor of `ffsvc22_dev_043388`
- keep `ffsvc22_dev_063743` quarantined in favor of `ffsvc22_dev_063782`
- do not reintroduce quarantined rows into active manifests or trials without a
  fresh audit and an explicit canonical replacement

## 4. `keep`: quiet recordings stay in the corpus, but normalization must be explicit

Evidence:

- `67,943` rows are below `-30 dBFS`
- mean loudness is `-44.38 dBFS`
- the quality flags split as `very_low_loudness=64198`, `low_loudness=3745`

Decision:

- this is treated as part of the robust setting, not as a cleanup-only defect
- `KRYP-023` should implement bounded loudness normalization or gain staging
- experiments must log whether normalization is enabled

## 5. `keep`: silence-heavy rows stay in the corpus, with explicit VAD ablations

Evidence:

- `68,349` rows have at least `20%` silent windows
- `51,312` rows are in `high_silence_ratio`
- silence-ratio `p95` is `100%`

Decision:

- this is also part of the robust setting, not something to delete away
- `KRYP-022` should compare `no VAD`, `light trimming`, and `aggressive trimming`
- silence assumptions must stay explicit in preprocessing benchmarks and model
  comparisons

## Stop Rules

- Do not run baseline, verification, or threshold tuning against official dev
  trials until the `trial_reference_missing` issue is removed.
- Do not let feature extraction or augmentation depend on implicit backend
  resampling.
- Do not reintroduce rows from `quarantine_manifest.jsonl` into active manifests
  or speaker-disjoint trials.
- Treat quiet and silence-heavy rows as retained data slices that require
  explicit preprocessing choices, not silent filtering.
